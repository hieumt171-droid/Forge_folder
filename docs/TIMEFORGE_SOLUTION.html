TimeForge — Solution Document

Intelligent Time Tracking & Timesheet trên Jira Cloud · Forge · Forge SQL · Bảo vệ đồ án

1. Executive Summary

1.1 Mục tiêu kinh doanh & bài toán cần giải quyết

Bối cảnh: Công ty dùng Jira quản lý task hàng ngày. Cuối tuần/tháng cần biết mỗi người làm bao nhiêu giờ, loại việc gì (code, họp…), gắn issue nào — để báo cáo nội bộ hoặc tính chi phí project.

Hiện trạngVấn đề
Jira worklog (có sẵn)Chỉ ghi giờ đơn giản — không phân loại, không có bảng tuần, không duyệt, export khó
Tempo (app trả phí)Đủ tính năng nhưng ~$8–10/người/tháng — SMB 10–25 người khó chi trả
Clockify, TogglApp ngoài Jira — dev hay quên log, sai issue
Mục tiêu kinh doanh TimeForge: App trên Jira giúp SMB ghi giờ theo category, xem timesheet tuần, nộp/duyệt, export CSV — chi phí thấp hơn Tempo, không rời workflow Jira.

1.2 Tóm tắt giải pháp đề xuất (2–3 dòng)

TimeForge là Forge app trên Jira Cloud: Issue Panel ghi giờ nhanh, Global Page xem timesheet tuần, Project Page (giai đoạn 2) cho manager duyệt. Dữ liệu lưu Forge SQL. Một app cài trên nhiều Jira site — mỗi site dữ liệu tách riêng.

1.3 Lợi ích kỳ vọng & Scope

Lợi íchGiải thích ngắn
Triển khai nhanhForge host sẵn — không mua server; pattern đã có từ bookmark manager, stale report
Dễ mở rộngThêm module Jira (projectPage, adminPage) trong manifest, không đổi lõi SQL
Chi phí vận hành thấpKhông EC2/DevOps; billing theo usage Forge

Scope & phạm vi áp dụng

Multi-site1 site pilot → nhiều site cài cùng app (Marketplace)Cross-product (Jira ↔ Confluence)

Cross-product trong sườn solution = không áp dụng; đồ án tập trung Jira-only.

3. Solution Overview

3.1 Kiến trúc tổng quan

App gồm 3 tầng — dễ hình dung:

👤 User trên Jira → 🖥️ Giao diện
Issue Panel, Global Page → ⚙️ Resolver
hàm backend → 💾 Forge SQL

Resolver đôi khi gọi Jira API để kiểm tra issue có tồn tại không.

3.2 Lý do chọn Forge (so với Connect / native)

Connect (cũ)Jira nativeForge ✅
HostingTự thuê server—Atlassian host
MarketplaceĐang ngừng—Hướng chính thức
DatabaseTự quảnChỉ worklogForge SQL
Multi-siteTự code—forge install tự tách data

3.3 Mô hình triển khai: Single app — Multi-site installation

TimeForge (1 codebase, 1 appId)
  ├── Cài trên site-a.atlassian.net  → database riêng của site A
  ├── Cài trên site-b.atlassian.net  → database riêng của site B
  └── Pilot: hieumt-forge-resource.atlassian.net
        ├── development  (code mới, test)
        ├── staging      (QA)
        └── production   (demo bảo vệ)
Multi-site nghĩa là gì? Cùng một app TimeForge, nhiều công ty cài trên Jira site của họ. Mỗi lần cài = dữ liệu tách hẳn — site A không đọc được giờ của site B.

3.4 Luồng tương tác chính (Jira ↔ Forge app)

Trong scope đồ án chỉ có Jira + Forge — không có Confluence.

  1. User mở issue HSF-12 trên Jira
  2. Panel TimeForge hiện (do Forge render)
  3. User bấm Lưu → invoke('logTimeEntry') → Forge chạy resolver
  4. Resolver lấy accountId user → validate → gọi Jira API check issue → ghi SQL
  5. Trả kết quả → panel cập nhật danh sách giờ

4. Thiết kế kỹ thuật chi tiết

4.1 App manifest & module structure

modules:
  jira:issuePanel:
    - key: timeforge-panel
      resource: issue-panel
      resolver: { function: resolver }
      title: TimeForge

  jira:globalPage:
    - key: timeforge-timesheet
      resource: timesheet
      resolver: { function: resolver }
      title: My Timesheet

  jira:projectPage:          # Phase 2
    - key: timeforge-team
      resource: team
      resolver: { function: resolver }
      title: Team Timesheet

  sql:
    - key: main
      engine: mysql

  function:
    - key: resolver
      handler: index.handler

permissions:
  scopes:
    - storage:app      # cho Forge SQL
    - read:jira-work   # đọc issue để verify

confluence:contentBylineItem — không dùng trong đồ án này.

4.2 Data model & Storage strategy

MVP: chỉ Forge SQL — một nơi lưu, dễ hiểu. KVS/cache xem xét sau khi cần tối ưu.

Bảng time_entries — mỗi lần ghi giờ = 1 dòng

CộtVí dụÝ nghĩa
iduuidKhóa chính
account_idtừ Jira contextAi log — không cho user giả mạo
issue_keyHSF-12Issue nào
project_keyHSFProject (lấy từ issue)
categorydevdev | review | meeting | ops
duration_min90Số phút
logged_atngày làm việcĐể gom theo tuần
statusdraft / submitteddraft = sửa được; submitted = đã nộp

Bảng week_submissions — tránh nộp tuần 2 lần

Primary key: (account_id, week_start) — mỗi user mỗi tuần tối đa 1 dòng.

Entity isolation per site

Mỗi lần forge install trên một Jira site, Forge tạo database riêng. App không cần cột site_id — platform đã tách sẵn.

Indexing

Index (account_id, logged_at) — query timesheet theo tuần nhanh.

Cross-product integration

Không áp dụng — scope Jira-only. Toàn bộ module Jira dùng chung resolver + chung SQL.

4.3 Permission & Security model

Chủ đềCách làm
OAuth scopesstorage:app + read:jira-work — ít quyền nhất có thể
User contextLuôn dùng context.accountId từ Forge — không tin payload client
Isolation per siteMỗi installation = SQL namespace riêng
OwnershipMọi UPDATE/DELETE có WHERE account_id = ?
Forge sandboxFrontend không truy cập DOM Jira trực tiếp

External service

MVP không có backend riêng. Chỉ gọi Jira API qua @forge/api. Dev debug: forge tunnel (local). Production: code trên cloud Forge.

Feature flag per site

forge variables set --environment production FEATURE_APPROVAL=false
forge variables set --environment production APP_VERSION=1.0.0

4.4 Error handling & logging

Mỗi resolver bọc logger.run() — log khi bắt đầu, thành công, lỗi + thời gian chạy (durationMs). Pattern đã dùng ở bookmark manager, verify production bằng forge logs.

Input sai (issue key hsf-1, cursor sai kiểu…) → validation throw message rõ → user thấy lỗi, log ghi đủ debug.

5. Chiến lược Multi-site & Scalability

5.1 Site isolation model

Site A và Site B cùng cài TimeForge nhưng hai database độc lập. Lỗi hoặc data site A không ảnh hưởng site B.

5.2 Cấu hình per-site

ConfigLưu ở đâu
Feature bật/tắtForge environment variables
Budget projectBảng SQL project_budgets
Custom categoryBảng config (phase 2)

5.3 Onboard site mới

  1. Khách cài app từ Marketplace (hoặc forge install)
  2. Forge tạo SQL + quyền tự động
  3. Migration chạy tạo bảng
  4. Site cũ không downtime

5.4 Scaling & rate limit

Tình huốngXử lý
Nhiều user log cùng lúcForge tự scale function; resolver không giữ state trong RAM
Jira API trả 429Retry vài lần
Timesheet chậm (sau này)Index SQL; có thể thêm cache (optimization)

6. Deployment & CI/CD Pipeline

6.1 Environment strategy

EnvMục đích

6.2 Forge CLI workflow

forge deploy --environment staging --non-interactive
forge install --site hieumt-forge-resource.atlassian.net --product jira -e staging
forge deploy --environment production --non-interactive
forge logs --environment production --since 15m

6.3 CI/CD (GitHub Actions)

Push code → ESLint → Jest → build frontend → deploy staging → (approve) → deploy production. Pattern từ hieumt_issue_panel_bookmark_manager: ci.yml, cd.yml.
